# Project Architecture

## Architecture Philosophy

BetterCF follows a **microkernel architecture**. The core (`index.js`) is a minimal orchestrator that loads, configures, and manages independent feature modules. Each module is a plugin that knows nothing about other modules.

```
index.js (orchestrator)
 ├── env/         (kernel — config, environment)
 ├── helpers/     (kernel — shared utilities)
 ├── ext/         (plugins — feature modules)
 └── common.css   (kernel — base styles)
```

## What the Core Does

The orchestrator in `index.js` has exactly three responsibilities:

1. **Load configuration** — Read from `localStorage`, merge with defaults, sync from `chrome.storage.sync` (extension only).
2. **Install modules** — Iterate the modules array in order, calling each module's `install()`. Wrap in `tryCatch` so one failure does not block others.
3. **Register config callbacks** — For modules with a config key, subscribe to config changes so `install()`/`uninstall()`/`install(value)` are called reactively.

Nothing else belongs in the core. New cross-cutting concerns (e.g., analytics, error reporting) should be added as utility improvements, not core bloat.

## What Stays Modular

### Feature Modules (`src/ext/`)

Every feature is a standalone module. This must never change. The contract is:

```javascript
export const install = env.ready(function() { ... });
export function uninstall() { ... }
export function init() { ... } // optional, idempotent
```

Modules are gated by:
- **Config boolean**: Module is installed/uninstalled when config toggles.
- **Config number**: Module is reinstalled with new value when config changes.
- **Always-installed**: Module has no config key and runs on every page.

### Environment Layer (`src/env/`)

Provides platform abstraction. Currently supports extension (`env-extension.js`), userscript (`env-userscript.js`), and shared (`env-shared.js`). Future targets (e.g., Edge-specific behavior) should be added here.

### Helpers (`src/helpers/`)

Shared utilities that modules depend on. Currently:
- `dom.js` — DOM query, creation, event binding
- `events.js` — In-memory event bus
- `Functional.js` — FP utilities (pipe, map, once, safe, debounce)
- `util.js` — Utility functions

New helpers should be added here only when used by at least two modules.

### Styles (`src/common.css`)

The shared stylesheet is injected via the manifest in extension mode and via JavaScript in userscript mode. All feature-specific CSS is in `common.css` using prefixed class names.

## What Must Not Be Modular

The following must stay in the core:
- **Config lifecycle** (load, save, sync, propagate)
- **Feature installation loop** (import all modules, iterate, install)
- **Error isolation** (tryCatch per module install)
- **Version update notification**

These are cross-cutting concerns that every module depends on. Duplicating them into modules would violate DRY.

## Future Architecture Goals

| Goal | Rationale |
|------|-----------|
| **Dynamic module loading** | Only load modules needed for the current page. Reduces startup time on non-problem pages. |
| **Module-level CSS** | Each module's CSS co-located with its JS, extracted at build time. Easier maintenance. |
| **Build-time module registry** | A JSON manifest of all modules, their config keys, dependencies, and page targets. Generated by Rollup. |
| **Declarative module metadata** | Each module exports metadata (name, config key, page pattern, dependencies) instead of wiring them in `index.js`. |

These are aspirational. No changes to the current architecture should be made without a clear migration path that preserves backward compatibility.
