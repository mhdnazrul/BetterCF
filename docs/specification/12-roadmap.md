# Roadmap

## Near-Term Goals (0–6 months)

These are achievable with the current codebase and minimal architectural change.

| Priority | Goal | Effort | Depends On |
|----------|------|--------|------------|
| P1 | **Fix known bugs**: twin standings not removed on toggle, finder Escape handling, config sync race on first install | Small | None |
| P1 | **Add CHANGELOG.md** and adopt Keep a Changelog format | Small | None |
| P2 | **Remove `codeforcespp.png` and `cf++ logo.png` leftover assets** from Git history | Small | None |
| P2 | **Adopt commit convention** (Conventional Commits) for all future commits | Small | None |
| P2 | **Extract `formatShortcut` from Functional.js** into its own module to break the circular dependency noted in the source | Small | None |
| P3 | **Add `AbortController` support to `dom.on()`** for easier event cleanup in `uninstall()` | Medium | None |
| P3 | **Replace `.then()` chains with `async`/`await`** in `show_tutorial.js` and `twin.js` | Medium | None |
| P3 | **Add unit tests for `helpers/Functional.js`** using `uvu` or `node:test` | Medium | None |

## Mid-Term Goals (6–18 months)

These require moderate architectural changes but no fundamental rewrites.

| Priority | Goal | Effort | Depends On |
|----------|------|--------|------------|
| P1 | **Dynamic module loading** — Only load modules needed for the current page. Reduce startup time on non-problem pages. | Medium | None |
| P2 | **Module-level CSS** — Co-locate each module's CSS with its JS file, extract at build time. Replace monolithic `common.css`. | Medium | Dynamic module loading |
| P2 | **Build-time module registry** — A JSON manifest of all modules, their config keys, dependencies, and page targets. Generated by Rollup. | Medium | None |
| P3 | **Unit test suite for all feature modules** — Mock DOM, test install/uninstall cycles. | Large | Module-level CSS, dynamic loading |
| P3 | **Per-module error reporting** — Catch, format, and optionally report module installation errors with context. | Medium | Build-time module registry |
| P3 | **Accessibility audit** — Run axe-core on all BetterCF-created UI. Fix all critical and serious violations. | Medium | None |
| P3 | **CI/CD pipeline** — GitHub Actions for lint, test, build on every PR. | Small | Unit tests |
| P3 | **Performance regression CI** — Track bundle size and init time per commit. | Small | CI/CD pipeline |

## Long-Term Goals (18–36 months)

These are aspirational and may require significant investment.

| Priority | Goal | Effort | Depends On |
|----------|------|--------|------------|
| P2 | **Manifest V3 migration** — Redesign MPH to work with service workers. Evaluate `offscreen` document API for storage access. | Large | Build-time module registry |
| P2 | **Declarative module metadata** — Each module exports an object with its name, config key, page patterns, and dependencies. The core uses this metadata instead of the hardcoded `modules` array. | Large | Build-time module registry |
| P3 | **Integration test suite** — Puppeteer-based tests that load a headless Codeforces-like page and verify behavior. | Very Large | All mid-term goals |
| P3 | **Visual regression testing** — Screenshot comparison for popup, modals, navbar. | Medium | Integration tests |
| P3 | **Edge extension support** — Test and certify on Microsoft Edge. | Small | Manifest V3 migration (Edge requires MV3) |
| P3 | **User-contributed module system** — Allow users to write and load their own modules without modifying the core extension. | Very Large | Declarative module metadata, CI/CD |
| P4 | **Theme marketplace** — Allow users to share and install custom CSS themes. | Very Large | All of the above |

## Future Ideas (Not Yet Scheduled)

These are exploratory. They may be promoted to the roadmap after community discussion.

- **Verdict notification**: Desktop notification when a submission is judged.
- **Contest countdown**: In-navbar countdown to the next Codeforces round.
- **Problem rating preview**: Show problem rating in the problemset list without hovering.
- **Submission history graph**: Inline chart of recent submission verdicts on the user profile.
- **Custom shortcut profiles**: Named sets of shortcuts that can be switched between.
- **Standings heatmap**: Color-coded problem grid showing which problems were solved by how many participants.
- **Split-pane editor**: Side-by-side problem statement and code editor (requires significant new UI).

## How the Roadmap Is Maintained

- The roadmap lives in this document and is updated after every MAJOR or MINOR release.
- Priorities are set by the maintainer based on community feedback, bug frequency, and development effort.
- Community members are encouraged to pick up unassigned goals and submit pull requests.
- A goal that has had no activity for 12 months may be demoted or removed.
- New ideas are added to "Future Ideas" after a brief discussion in the issue tracker. They are promoted to the roadmap when someone volunteers to implement them or when they receive enough community support.
